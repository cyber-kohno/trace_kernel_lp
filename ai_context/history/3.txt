＞実行環境（エンジン）と標準APIについて
→
Trace KernelのTypeScript実行環境は、
WebWorkerになります。
そのため、ブラウザのJSで使えるAPI（DomParserなど）は使えません。

メインのUIスレッドで実行ボタンを押下し、
プログレスバーなどはUIスレッドで描画しますが、
裏で非同期で動いているWebWorkerからメッセージングを受けて、
状態をUIに反映しています。


＞外部ライブラリ（npm・サードパーティ）の扱い
→
外部ライブラリの利用は一切できません。
また、1ワーク1プログラム（1ソース）という制約があり、
処理ごとにファイル分割して、インポートして整頓するようなこともできません。
Trace Kernelは小さな問題を道具で解決するという理念があり、
例えば合計1000行程度に及ぶ大きなプログラムを書く場合は、
Trace Kernelの範疇を超えていると判断します。

その代わりに、
好きな言語で、
自由に大きなプログラムやサードパーティライブラリが必要な処理を実装し、
$process機能で連携すれば、
例えば、大量のmp3ファイルのメタ情報を修正する
などのニッチな業務もTrace Kernelを介して実現できます。

Trace Kernelでは最低限必要なハブとしてのAPIと、
実行環境を提供します。


＞パフォーマンスやメモリの上限・制限事項
→
データセットは10万件程度のファイル読み込みは耐えられることを確認していますが、
PC性能に依存する部分もあるので、
上限の具体的な境界は確認していません。
リソースべた貼りは、1万件程度でかなり重くなります。
純粋なhtmlのテキストエリアを使っているので、
巨大なファイルは$fs.readText()で動的にメモリにロードすることが推奨されるかもしれません。

$printlnでのGUIストリームの出力に関してのみ、
特別な対応をしています。

アプリはTauri（バックエンドがRust）を採用しているので、
メインスレッドにストリームのデータは一切持たず、
$printlnなどでストリーム書き込みした文字列は、
全てRust側のメモリで保持しています。
画面（スクロール範囲内）に表示する内容のみ、
常にRust側からフェッチして表示するので、
70万行出力してもパフォーマンス維持することを確認しています。


＞トランザクション機能（Confirm -> Verify）の競合について
＞仮想FS上で変更を積み、最後のダイアログで「Commit」するまでの間に、
＞別の外部アプリが対象ファイルの中身を書き換えてしまった場合などは、
＞Verifyフェーズで弾かれる（エラーになる）仕様でしょうか？
→
はい。その通りです。
スクリプトのランタイム時には存在していたファイルが、
トランザクションの検証フェーズ時に、消されていた場合などは、
エラーになり実行できません。

＞データセットの Runtime auto でのプレビュー
Runtime autoにすると、スキャンはランタイム時に委ねられるので、
GUIで対象確認などの恩恵は一切受けられません。
自動的にHITした全てのファイルが処理されます。

データセットでかなり複雑なフィルター条件を作って、
Runtime autoで処理したいけど、
条件が正しいか心配で、事前に対象だけチェックしておきたい
ということであれば、
チェックするためだけのワークを追加して、ループ回して$printlnするだけの処理を書く必要があります。

ただ、自分なら一次的にDirect Chooseに変更して、
GUI上でスキャンしてみてちゃんとHITすることをGUIで確認した後に、
Runtime autoに戻します。
