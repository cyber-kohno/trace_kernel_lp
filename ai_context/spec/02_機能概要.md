# 機能概要

## 画面の全体構成

- スタート画面
- ワークスペース管理画面
- プログラムコードエディタ（Monaco）
- トランザクションダイアログ

### スタート画面

![スタート画面](./screen_shot/スタート.JPG)

アプリを起動した直後はこの画面を表示します。
この画面はワークスペースを開く際に、新規で始めるか、既存のファイルから読み込んで始めるかを選択するために存在します。
ワークスペースを起動することで、作業が可能な状態となります。

- **Start Empty**：新規に始める
- **Open File**：既存ファイルを読み込んで始める

### ワークスペース管理画面

![ワークスペース管理](./screen_shot/ワークスペース管理.JPG)

ワークスペースを開くと必ずこの画面に遷移します。

1. **メニューヘッダー**
2. **左半分**：コンテキストツリー
3. **右半分**：ツリーで選択中の詳細編集画面

左半分のコンテキストツリーには、ワークスペースで管理する5つの要素が存在します。
（`env`, `resource`, `dataset`, `process`, `work`）

いずれも、＋ボタンから要素を追加することができ、名前をつけて複数持つことができます。
この中で、`work`だけが他とは違い、プログラムを書いて実行する**作業単位**です。
`work`以外の要素は、`work`のプログラムで参照する**コンテキスト情報**となります。

- **`context`**
  - `env`
  - `resource`
  - `dataset`
  - `process`
- **`program`**
  - `work`

各要素の＋ボタンを押下すると、要素が追加されます。

![要素追加](./screen_shot/要素追加.JPG)
（画面イメージはresourceを追加した例）

追加した要素を選択すると、右半分の詳細エリアに編集画面が表示されます。
要素の編集に「保存」の概念はなく、編集すると即座にワークスペースに反映されます。

- 要素は必ず名前を持っており、名前が重複した場合は両方がエラー（`error`）となります。
- 必須項目が未入力であったり不整合がある場合は赤色の `error` となります。
- 情報がすべて正常であれば緑色の `enable` となります。

#### コンテキスト要素の利用例（ワークスペース設定例）

```text
- env
  - DEST_DIR
- resource
  - userData
  - setting
- dataset
  - workspace
- process
```

上記のように定義していた場合、以下のように `work` で書くプログラム内で利用できるようになります。

```javascript
$env.DEST_DIR
$resource.userData
$resource.setting
$dataset.workspace
```

具体的には、Monacoエディタに情報が注入され、TypeScriptのエディタで補完情報として参照できるようになります。
エラーが存在するコンテキスト要素は、プログラム内からは利用できません。

### ワークスペースの保存について

画面上部の「Save」ボタンを押下するか、`Ctrl+S`キーでワークスペースをファイルに保存することができます。
保存する拡張子は `.trk` で、中身はワークスペースのJSONをgZip圧縮したプレーンテキストです。
一度保存した後はファイル名の表示となり、上書き保存となります。

## コンテキスト要素

### `env` について

変数名と値のペアを登録し、変数名から値を取得することができるシンプルな要素です（環境変数用途）。

たとえばファイル出力を行う処理を書く場合、出力先のディレクトリを直接プログラムに書くのではなく `env` で定義しておくことで、エスケープミスを防ぎ、環境ごとのメンテナンスを容易にします。

![env追加時](./screen_shot/env追加時.JPG)

変数名を `DEST_DIR` と定義した場合、スクリプト上では以下のように参照できます。
```javascript
tx.saveFile(`${$env.DEST_DIR}\\${fileName}`, content);
```

また、`env`で定義した変数は、他のコンテキストのパス項目で `%` で囲うことで変数展開して参照することもできます（例: `%WORKSPACE%\testApp\src\`）。

- **name**: 変数名（大文字のみ許容）
- **value**: 値
- **purpose**: 用途（パスの存在チェックなどに利用）

### `resource` について

静的なリソースを変数として割り当てる要素です。解析対象のソース（ログやCSVなど）を登録するために利用します。

![resource追加時](./screen_shot/resource追加時.JPG)

- **variable_name**: 変数名
- **source**: ソース（複数行テキストを入力可能）
- **parse_method**: ソースの変換方法

テキストエリアにCSVなどを直接貼り付けて割り当てます。
ソースがテーブルデータ（CSVまたはTSV）の場合、`parse_method` を指定することで、プログラムから呼び出す際に自動でオブジェクトとして扱うことができます。

```javascript
$resource.userData.forEach(user => {
  // ヘッダが'name'の各レコードの値を参照
  console.log(user.name);
});
```

**`parse_method` について**
- CSV to JSON
- TSV to JSON

> **[!] Tips**: GUIのテキストエリアを使用している都合上、数万行レベルの巨大なCSV等を直接貼り付けると動作が重くなる場合があります。大容量のデータはここで定数として定義せず、プログラム内で `$fs.readText()` を使ってファイルをフルパスから直接動的に読み込むアプローチを推奨します。

### `dataset` について

ルートパスから相対パスの集合を管理します。動的リソース（ファイル群の一括処理）を遅延ロードするために利用します。

【用途例】
- 大量ファイルのリネーム
- 大量ログを解析
- パターン検出など

![dataset追加時](./screen_shot/dataset追加時.JPG)

- **variable_name**: 変数名
- **root_path**: ルートパス
- **encoding**: ファイルを開く際のエンコード（utf8 / sjis）
- **scan_phase**: スキャンタイミング（`Runtime auto` / `Direct choose`）

#### `Runtime auto`
プログラム実行時の前処理でルート配下のファイルを検索し、全てのファイルを相対パスとして取得します。

> **[!] Tips**: 複雑な抽出条件を設定し、意図通りにファイルがHITするか不安な場合は、一時的に「Direct choose」に切り替えて「Scan」ボタンを押すことで、対象ファイルをGUIでプレビュー確認できます。確認後「Runtime auto」に戻すことで、安全に自動化フローを構築できます。

#### `Direct choose`
GUI上で対象の相対パスを直接選択して固定します。

![抽出対象パネル](./screen_shot/抽出対象パネル.JPG)

対象ファイルをフィルターする条件が設定できます。
- **limit_depth**: スキャン階層数の制限
- **directory_filter_conditions**: ディレクトリの抽出条件（階層ごとに除外・内包指定）
- **file_filter_conditions**: ファイルの抽出条件

![ディレクトリ条件追加時](./screen_shot/ディレクトリ条件追加時.JPG)
![ファイル条件追加時](./screen_shot/ファイル条件追加時.JPG)

条件を設定した後、画面右下の「Scan」ボタンを押下して走査します。
![Scanボタン表示](./screen_shot/Scanボタン表示.JPG)
![スキャン実行時](./screen_shot/スキャン実行時.JPG)

検索が終わるとツリーが表示され、自由に選択できます。
![ファイル選択（ツリー表示）](./screen_shot/ファイル選択（ツリー表示）.JPG)
![ファイル選択（フォルダを展開）](./screen_shot/ファイル選択（フォルダを展開）.JPG)
![ファイル選択（選択状態の色分け）](./screen_shot/ファイル選択（選択状態の色分け）.JPG)
![ファイル選択（フラット表示）](./screen_shot/ファイル選択（フラット表示）.JPG)

選択後、「Transfer」ボタンを押下して確定すると、対象ファイルがリストアップされます。
![Transfer押下後](./screen_shot/Transfer押下後（リストアップ）.JPG)

#### プログラムでの利用方法
```javascript
for(const d of $dataset.workspace) {
  d.fileName     // ファイル名のみ取得
  d.absolutePath // フルパスを取得
  d.relativePath // 相対パスを取得
  await d.content(); // ファイルの中身のソースを非同期で取得（await必須）
}
```

### `process` について

外部プログラムを、スクリプト上で「関数」として呼び出して利用できる橋渡しの仕組みです。

![process追加時](./screen_shot/process追加時.JPG)

関数名と実行するプログラムのパスを設定することで、`$process.test()` の用に関数として呼び出せます。スクリプト引数とコマンドライン引数をマッピングすることも可能です。

![プロセス設定例](./screen_shot/プロセス設定例.JPG)

```javascript
const {stdout} = await $process.hello('taro');
$println(stdout); // "taroさんこんにちは"
```

## プログラム要素 `work` と独自 API

### `work` について

TypeScriptのプログラムを書いて実行するワーク単位です。
プログラムのファイル分割やインポート・エクスポートはなく、**「1ワークで1プログラム制約」**です。

![work追加時](./screen_shot/work追加時.JPG)
- **name**: ワーク名
- **output_method**: 出力方式（基本は「Plain」でテキストを出力）

「Open Editor」ボタンを押下すると全画面エディタ（Monaco）が表示され、動的にコンテキストが補完されます。
![プログラムエディタ表示時](./screen_shot/プログラムエディタ表示時.JPG)
![補完](./screen_shot/補完.JPG)

**実行環境の特性:**
ワークのプログラムは、バックグラウンドの **WebWorker環境で安全に非同期実行されます**。DOM等のブラウザAPIは使えませんが、UIをフリーズさせることなく実行可能です。

F5キー等で実行すると、出力結果パネルが表示されます。
![プログラム実行時（出力結果）](./screen_shot/プログラム実行時（出力結果）.JPG)

> **[!] パフォーマンスの強み**: Trace KernelはバックエンドにTauri(Rust)を採用しており、出力ストリームはUIスレッドに負荷をかけない設計になっています。そのため `$println` で70万行を超える大量のログを出力しても、エディタやGUIが重くなることなくスムーズに動作します。

### 独自API `$state`（プログレスバー・モニター）

実行に時間がかかる全件スキャン等の際、GUIにリアルタイムの進捗プログレスバーや固定文字を表示できる強力なAPIです。

![スクリプト実行中](./screen_shot/スクリプト実行中.JPG)

**プログレスバーの使用:**
```javascript
const {tick} = $state.useProgress($dataset.files.length);
for(const d of $dataset.files) {
  // 処理
  tick(); // プログレスバー加算
}
```
![スクリプト実行中（プログレスバー）](./screen_shot/スクリプト実行中（プログレスバー）.JPG)

**モニターの使用:**
固定表示で件数などを更新したい場合は `$state.useMonitor(数)` を宣言します。
```javascript
const [summary, current] = $state.useMonitor(2);
summary(`パターン一致件数： ${matchCnt}件`);
current(`解析中のファイル： [${d.fileName}]`);
```
![スクリプト実行中（モニター）](./screen_shot/スクリプト実行中（モニター）.JPG)

### 独自API `$fs` とトランザクション機能

ファイルシステムへの読み書きを行う関数群です（`exists`, `glob`, `stat`, `makeDir`, `deleteFile` 等）。

特筆すべきは仮想FSを利用した **トランザクション API** です。
スクリプト実行時に仮想のFS領域に対して論理操作（更新や作成）を積み、プログラム終了後に実FSへの一斉検証・コミットを行うことで、ファイル破壊のミスを徹底的に防ぎます。

```javascript
const tx = $fs.useTransaction();
const file = await tx.openText(`${DIR}\\targetFile.txt`, 'utf8');
tx.updateText(file.token, "新しいテキスト");
```

仮想FSへのオーダーが積まれると、スクリプト完了時に以下の「トランザクションダイアログ」が表示されます。

![トランザクションダイアログ](./screen_shot/トランザクションダイアログ.JPG)
![書き込み内容確認](./screen_shot/書き込み内容確認.JPG)
![更新内容確認](./screen_shot/更新内容確認.JPG)

ダイアログのフェーズ：
1. **confirm（確認フェーズ）**: 内容に問題がないかGUIで目視確認します。
2. **verify（検証フェーズ）**: 実装FSに反映可能かシステムがチェックします。
   - **※ スクリプト実行時には存在したファイルが、検証フェーズの間に外部の別アプリによって削除されてしまった場合、厳格な整合性チェックによってエラーとして弾かれ、実FSの破壊を防ぎます。**
3. **commit（処理済み）**: すべての検証をパスした場合のみ、実FSに安全に変更を反映します。

![書き込み完了](./screen_shot/書き込み完了.JPG)

このワークフローにより、「中途半端な状態で処理が止まる」といったファイル操作における事故を根絶します。
