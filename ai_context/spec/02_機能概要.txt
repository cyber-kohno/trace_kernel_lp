■ワークスペースの保存について
ワークスペース管理画面のヘッダメニューのSaveボタンを押下するか、
Ctrl+Sキーで、ワークスペースをファイルに保存することができる。
保存するファイルは、拡張子が.trkとなる。
中身はワークスペースのjsonをgZip圧縮したプレーンテキストである。

ワークスペースを新規で開いて、
一度も保存していない場合は、
アプリフレームのヘッダが(Untitled)*の表示となり、
保存時に、ファイル保存ダイアログを表示する。

一度保存した後は、
ファイル名の表示となり、保存時は上書きとなる。

既存のファイルを読み込んでワークスペースを開いた場合も、
上書き保存となる。


■コンテキスト要素「env」について
envは、変数名と値のペアを登録し、
変数名から値を取得することができるシンプルな要素である。
環境変数として利用することを目的としている。

例えば、プログラムの中でファイル出力を行う処理を書く場合、
出力先のディレクトリを、
プログラムに直書きするのではなく、
envのGUI上で定義しておくことで、
\のエスケープミス防止や、
環境が変わった際に、
プログラムを開かずともスムーズにメンテナンスできる。

変数名を「DEST_DIR」と定義した場合、スクリプト上では、
tx.saveFile(`${$env.DEST_DIR}\\${fileName}`, content);
のように参照することができます。

また、envで定義した変数は、
他のコンテキストのパスを扱う項目で、
%のブレースフォルダで囲うことで、
%WORKSPACE%\testApp\src\
のように参照することができます。

envの＋ボタン押下時は、以下の状態となる。
screen_shot/env追加時.JPG

・name：変数名（大文字のみ許容）
・value: 値
・purpose：用途

purposeは、用途がパスである場合は、
選択することで、
値に入力したパス（ファイル or ディレクトリ）が、
存在するかリアタイムでチェックできる。
この項目はあくまでチェック用で、
パスが不正（存在しない、ファイル・ディレクトリの不一致）であっても、
env要素としてはエラーにならない。


■コンテキスト要素「resource」について
resourceは、静的なリソースを変数で紐づけして割り当てするためのものである。
変数と値のペアを直接GUI上で管理する点以外は、
envと同じであるが、
resourceは、解析対象のソース（ログ・csvなど）を割り当てるために利用する。

resourceの＋ボタン押下時は、以下の状態となる。
screen_shot/resource追加時.JPG

・variable_name：変数名
・source：ソース（値）
・parse_method：ソースの変換方法

値割り当てでenvと違うのは、
複数行テキストである点のみ。

ここに、ログのテキストやcsvのテキストを、
直接貼り付けて、静的リソースとして割り当てる。

ソースがテーブルデータ（csv or tsv）の場合、
parse_methodを指定することで、
ソーステキストのヘッダから、カラム名を解析し、
プログラムから呼び出す際に、
オブジェクトとして扱うことができる。

例えば、userDataという変数名で、
csvを定義していた場合、
parse_method指定していなければ、
プログラムではstring型の純粋なテキストとして注入されるため、
要素の解析（改行、カンマで分割など）は手動となるが、
パースを指定していた場合は、
型付けされたオブジェクトとして最初から扱えるようになる。

$resource.userData.forEach(user => {
  user.name // ヘッダがnameの各レコードの値を直接参照できる
});

parse_method
・CSV to JSON
・TSV to JSON
テキストに合わせて任意の変換方法を選択する。

TSVの場合は、全ての列がstring型として変換されるが、
CSVの場合は、解析時に全てのレコードの値が、
NULLでなくダブルクォーテーションで囲われていない場合、
数値変換が可能であれば、number型として変換される。


■コンテキスト要素「dataset」について
datasetは、ルートパスとルートからの相対パスの集合を管理することができる。
resourceが静的リソースを扱うのに対して、
datasetは動的リソースを扱うために利用する。

PC内のファイルを遅延ロードすることができる。

用途としては、
・大量ファイルのリネーム
・大量ログを解析
・プロジェクト内のアンチ実装パターンの検出
など。


datasetの＋ボタン押下時は、以下の状態となる。

・root_path：ルートパス

・scan_phase：ルート配下のファイルをどのタイミングでスキャンするか
→

全てのファイルを相対パスとして取得する。

ただし、後述するフィルター条件を指定した場合、
条件に一致するファイルのみ対象とする。

Direct chooseを選択した場合、
GUI（datasetの詳細画面上）で、直接相対パスを選択する。

■コンテキスト要素「process」について
processは、外部のプログラムを、
Trace Kernelのスクリプト上で、
関数として呼び出して利用できるようにする、
橋渡しの仕組みを提供する。

関数名と実行するプログラムのパスを設定することで、
$process.test()のように関数として呼び出すことができる。

プロセスの管理では、コマンドライン引数を指定することが可能で、
実行するプログラムに引数を与えて実行することができる。
また、スクリプト引数を定義し、
コマンドライン引数に注入することで、
スクリプトで実行時に関数の引数を経由して、
任意のコマンドライン引数を与えることも可能。

スクリプトからプロセス経由で、
外部プログラムを実行すると。
実際にexeなどを直接叩いた時と同じように、
即処理が行わるが、

プログラムが標準出力を行う場合は、
stdout, stderrをスクリプトで取得することができる。
（プログラムの実行は非同期なのでawait必須）

processの＋ボタン押下時は、以下の状態となる。
screen_shot/process追加時.JPG

・function_name：関数名
・program_path：プログラムのフルパス
・script_argument_defs：スクリプト引数
・command_argument_values：コマンドライン引数
・timeout_millisecond：タイムアウトまでの時間（ミリ秒）
・response_encoding：stdout,stderrそれぞれにエンコードを指定

script_argument_defsは、
＋ボタンを押下することで、スクリプト引数を追加する。
追加すると、スクリプトで関数呼び出し時に引数を求められる。

スクリプト引数の定義は、
変数名のみの指定で良いが、数値を指定したい場合は、
右隣の「Number」を押下してONにすることで、
スクリプトで実装時の引数の型をnumber型にすることができる。
（数値以外はエラーとなる）
左の－ボタンで削除できる。

スクリプト引数は変数名を__（アンダースコア２つ）で囲った、
ブレースフォルダで、後述するコマンドライン引数に注入することができる。
コマンドライン引数に注入しない限り、
スクリプト引数の定義だけしても、プログラム実行には反映されない。

command_argument_valuesは、
＋ボタンを押下することで、実行するプログラムのコマンドライン引数を追加する。
必要な項目は引数の値のみである。
左のーボタン押下で削除できる。

スクリプトで呼び出して、timeout_millisecondで指定した時間を超えると、
ランタイムエラーとなる。

例として、JavaScriptで、
”〇〇さんこんにちは”と出力するだけのプログラムを、
（名前の部分はコマンドライン引数）
node.jsを用いて実行する場合の設定例は以下のようになる。
screen_shot/プロセス設定例.JPG

ポイントとして、実行するプログラムは、
nodeのランタイムなので、node.exeとであり、
nodeで実行するJSのプログラムファイルが第一引数となる。
JSプログラムの引数は、定義上は第二引数となる。

スクリプトでは、以下のように利用する。
const {stdout} = await $process.hello('taro');
$println(stdout); // "taroさんこんにちは"


実用例としては、
本番環境で稼働しているJavaプログラムモジュールに対して、
脆弱性の含まれたライブラリが利用されていないかチェックしたいが、
モジュールは数10個のjarで固められており、
中にclassファイルが入っている。
全てのclassファイルを逆コンパイルして、
該当のライブラリが含まれていないかgrep検索する必要がある。

上記のような場合に、
datasetでモジュールの全てのjarを対象にし、
jarを解凍する外部プログラムで、一括解凍する。
解凍したclassファイル群を、別のdatasetで割り当てし、
classファイルを逆コンパイルして、文字列検索し、
対象のプログラムをリストアップする。

このようなことが可能となる。


■プログラム要素「work」について
workは、TypeScriptのプログラムを書いて、
実行するワーク単位として扱う。
プログラムのファイル分割や、インポート・エクスポートはなく、
1ワークで1プログラムとなる。

ワークの用途としては、
・ワーク1：データの収集→出力結果を中間データとしてリソースに保存（手動）
・ワーク2：データの解析、整形→〃（中間データ保存）
・ワーク3：ファイル出力
のように、フェーズを分けて個別に実行していくことを想定している。

プログラムでは、$（ドルマーク）で始まるAPIにより、
様々な独自プログラムと、注入したコンテキストを利用することができる。

コンテキストについては、前述で説明したが、
注入されるAPIの代表として、
$print
$println
がある。
これは、console.logとは異なり、
Trace KernelのGUI上の出力パネル（本来であればターミナル）に、
文字列を出力することができる。

その他
・$fs：ファイルシステムを扱う
・$parse：Excel,XML,CSVなどのパースを行う
・$net：htmlのソース取得や、外部APIを叩く
などがある。

workの＋ボタン押下時は、以下の状態となる。
screen_shot/work追加時.JPG

・name：ワーク名
→名前を設定するが、コンテキストと違い、
　他所で参照されることはないため、ワークの識別名でしかない。
・output_method：出力方式
→基本は「Plain」$print（$println）でシンプルに文字列出力ができる。
　他の方式を選ぶと$printは注入されず、
　別の方法で出力する。
　例）Channelを選択すると、
　チャンネルストリームをスクリプト上で動的に生成して、
　複数ストリームに対してテキストやテーブル形式で出力できる。
・api_injections：注入される独自API
→output_methodにより変わる
・context_injections：注入されるコンテキスト
→注入可能なコンテキストの一覧を表示する。
　コンテキスト定義していてもエラーがある場合は、
　ここに表示されない。

■独自API「$state」について
「$state」APIは、スクリプト実行時に、
GUIにプログレスバーや、モニター（固定文字）を表示することができる。

例えばデータセットで定義した数1000のファイルを、
順番にオープンして解析していく処理において、
実行完了までに5分ほどかかる場合、
実行中は「Executing...」の表示しかされない。
screen_shot/スクリプト実行中.JPG

いつ終わるのか、今どこを解析しているのか、
$printで出力しないと状況がわからない。

そこで、このAPIを用いることで簡単にプログレスバーを表示できる。
$state.useProgress(totalCount)
を実行することで、指定の母数のプログレスバーを、
GUIに表示することを宣言し、
操作するためのハンドルを取得できる。

ハンドルのプロパティであるtick()関数で、
プログレスを1つ進めることができる。
使い方は以下のようになる。

const {tick} = $state.useProgress($dataset.files.length);
for(const d of $dataset.files) {
  // 処理
  tick(); // プログレスバー加算
}

このように、
・プログレスバーの宣言
・加算関数呼び出し
の2行を追加するだけで、
簡単にプログレスバーを実行中の画面に表示できる。
screen_shot/スクリプト実行中（プログレスバー）.JPG


また、条件に一致したファイルの件数（実行中にリアルタイム更新）や、
解析中のファイルを、実行中に確認したい場合は、
モニターを使えば簡単に実現できる。

モニターは、$state.useMonitor()関数で宣言し、
引数で利用するモニター数を指定することで、
指定した数だけ、モニターの更新関数を配列で返すようになっている。

例えば、条件に一致した合計数と、解析中のファイル名を、
モニターで画面に固定表示（リアルタイムで更新）したい場合、
以下のように2つ確保し、分割代入で更新関数を取得する。
const [summary, current] = $state.useMonitor(2);

取得した関数は、
summary(`パターン一致件数： ${matchCnt}件`);
current(`解析中のファイル： [${d.fileName}]`);
のように関数として実行することで、画面に文字列を取得する。

■独自API「$fs」について
ファイルシステムへの読み書きを行う関数群を提供する。

$fs.exists(`${DIR}\\targetFile.txt`)
のように各APIを呼び出せる。

exists: (path: string) => Promise<boolean>;
→パスの存在チェック
glob: (pattern: string) => Promise<string[]>;
→パターンのパス一覧を取得
stat: (path: string) => Promise<FileStat>;
→作成日、更新日、ファイルサイズを取得
readDir: (dir: string) => Promise<{ name: string; isDir: boolean; }[]>;
→ディレクトリを読み込み、配下の要素一覧を返す
readBinary: (filePath: string) => Promise<ArrayBuffer>;
→バイナリを読み込む
readText: (filePath: string, encoding: 'utf8' | 'sjis') => Promise<string>;
→テキストを読み込む
saveText: (filePath: string, content: string) => Promise<void>;
→テキストを保存（書き込み）
copyFile: (src: string, dest: string) => Promise<void>;
→ファイルをバイナリコピーする
makeDir: (dirPath: string) => Promise<void>;
→ディレクトリを作成（存在することを保証）
deleteFile: (filePath: string) => Promise<void>;
→ファイルを削除
deleteDir: (dirPath: string) => Promise<void>;
→ディレクトリを削除（中身が空の場合のみ可能）
renameFile: (targetFilePath: string, newFileName: string) => Promise<void>;
→ファイル名変更（同ディレクトリ内のみ）
renameDir: (targetDirPath: string, newDirName: string) => Promise<void>;
→ディレクトリ名変更（同ディレクトリ内のみ）


上記は非同期で即FSに読み書きをするのに対し、
仮想FS上に論理操作を積み、トランザクションで処理することを可能とする、

useTransaction: () => DclFSTransaction.TransactionAPI;
が存在する。

実行すると、トランザクション利用を宣言し、
仮想FSに書き込むためのハンドル（API群）を取得する。

const tx = $fs.useTransaction();

tx.openText(`${DIR}\\targetFile.txt`, 'utf8');
のように使える。

トランザクションAPIは以下が存在する。
書き込み系は処理のオーダーを積むだけなので、
同期処理で書けるのが特徴。

openText: (filePath: string, encorde: "utf8" | "sjis") => Promise<{ token: RuntimeUtil.FileToken; content: string; }>;
→
テキストを読み込み、中身の文字列とファイルトークンを取得する。
この関数を呼び出すと、指定したパスはチェックアウト状態となり、
トークン経由でしか、変更操作（更新・削除・リネーム）できない。

saveText: (filePath: string, content: string) => void;
→テキストファイルを新たに作成する。

updateText: (token: RuntimeUtil.FileToken, content: string) => void;
→テキストを更新する。openTextとペアで利用する。

copyFile: (from: string, dest: string) => void;
→ファイルをコピーする。

deleteFile: (filePath: string) => void;
→ファイルを削除する。

deleteFileByToken: (token: RuntimeUtil.FileToken) => void;
→トークン経由でファイルを削除する。

renameFile: (targetFilePath: string, newFileName: string) => void;
→ファイル名を変更する。

renameFileByToken: (token: RuntimeUtil.FileToken, newName: string) => void;
→トークン経由でファイル名を変更する。


トランザクションAPIはスクリプト実行時に、
論理整合性を担保する設計になっている。

オープンしたパスに対して、
直接ファイル操作（保存、削除、リネーム、再読み込み）を行った場合、
ランタイムエラーとなる。


トランザクションAPIの重要なポイントとして、
useTransactionは、トランザクションの開始ではなく、
利用することの宣言である。

useTransactionを呼び、ハンドル経由で仮想FSに書き込みオーダーを積むと、
プログラム実行終了時に、トランザクションダイアログを表示する。
screen_shot/トランザクションダイアログ.JPG

トランザクションダイアログでは、
以下の状態を管理している。
１．confirm：確認フェーズ
２．verify：検証フェーズ（変更可否のチェック）
３．commit：処理済み

１．confirmでは、
仮想FSのオーダー内容に問題がないか確認できる。
ファイル名を押下することで、変更内容を確認することができる。
screen_shot/書き込み内容確認.JPG
screen_shot/更新内容確認.JPG

問題ない場合、右下のコミットボタンを押下することで、
２．verifyに進む。

検証フェーズでは、
実FSをチェックし、変更可否のチェックを行う。
・ファイル作成しようとしたディレクトリが存在しない
・更新しようとしたファイルが存在しない
・書き込み権限がない
などの問題や不整合が発生した場合、
ここでエラーとなり、実FSの変更処理は行わない。

1つもエラーが発生しなかった場合、
実FSへの変更処理が実行される。
screen_shot/書き込み完了.JPG

・ランタイム：仮想FSにて論理整合を検証
・トランザクション：目視での認識合わせと、実FS変更可否を検証

このワークフローにより、
実装ミスにより意図していないFS変更や、
途中までFS変更が実行されて、
中途半端な状態で止まるといった問題を解消し、
安全で間違いのないFS操作を実現する。
