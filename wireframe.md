## ⚡ 2. 課題と解決（Why Trace Kernel?）
**レイアウト**: テキストとデモイメージを並列（ジグザグ配置の導入部分）。

**コピー**:
> **「ちょっとした処理を書く」までのハードルが高すぎる。**
> 
> ログファイルを一部抽出したい。数百のCSVデータをサクッとSQLに変換したい。
> ほんの数行のスクリプトで解決する問題なのに、わざわざ作業用のディレクトリを作り、TypeScriptの環境をセットアップし、ファイルを読み込むためのボイラープレートを書く...。
> 
> Trace Kernelなら、**アプリを開くだけで即座にコードを書き始められます。**  
> 「問題を解くこと」だけに集中できる環境を提供します。

## 🏗 3. ワークスペースとコンテキスト（Workspace & Context）
**レイアウト**: GUIのツリー画面の画像をベースに、ツリーの各要素（env, resource, dataset）を個別カードで解説。

**コピー**:
> **ファイル読み取りのコードはもう書かなくていい**
> 
> Trace Kernelでは、設定やリソースなどの前提情報（**コンテキスト**）と、処理本体（**ワーク**）を一つの**ワークスペース**内で集中管理します。
> GUI上で定義したコンテキストは、Monacoエディタに自動的に注入され、強力な型推論・補完とともに即座に呼び出せます。
>
> 用途に合わせて、以下の3つのコンテキストを使い分けられます。

### 🏷️ env（環境変数）
> **パスや設定値の定数化**  
> ファイルの出力先ディレクトリなどをGUI上で変数（例：`DEST_DIR`）として定義します。プログラムに直書きしないためエスケープミスを防ぎ、環境ごとのパス変更もGUIからワンタッチで行えます。（呼び出し例：`$env.DEST_DIR`）

### 📄 resource（静的リソース）
> **テキストやCSVを直接オブジェクト化**  
> GUIのテキストエリアにCSVやTSVを貼り付け、「CSV to JSON」パースを指定するだけ。実行時には自動でJSONオブジェクトの配列としてパースされるため、面倒な文字列分割を書くことなく、即座にループ処理へ移行できます。（呼び出し例：`$resource.userData`）

### 📂 dataset（動的データセット）
> **ローカルPC内の大量ファイルを安全にバインド**  
> 指定したディレクトリ配下から、複雑なフィルタ条件（例：「`.git`フォルダを除外し、`*.java`のみ抽出」）に合致するファイルだけをGUIで直感的に抽出。数千のファイル群でも遅延ロードで安全に走査・解析が可能です。（呼び出し例：`$dataset.workspace`）

## ⚙ 4. 強力な独自API（Trace Kernel APIs）
**レイアウト**: エディタ画面とGUI出力パネルの画像を交えて紹介。

**コピー**:
> **ターミナル不要の直感的なインターフェース**
> 
> ただのTS実行環境ではありません。GUIアプリならではの利便性を高める独自API群（`$`で始まる関数）が最初から組み込まれています。

- **`$print` / `$println`**: ターミナルを開くことなく、アプリ内の出力結果パネルに直接文字列を表示します。
- **`$state` によるプログレスバー表示**: 「いつ終わるかわからないループ処理」も、数行追加するだけでGUI上にリアルタイムなプログレスバーやモニター（解析中のファイル名表示など）を簡単に表示できます。
- **`$channel` によるテーブル出力**: 単なるテキストだけでなく、出力結果をチャンネルに分けて、整った表（テーブル）形式のストリームとして表示・整理することが可能です。

## 💻 5. 活用例（Use Cases）
**レイアウト**: 実用的なコードスニペットと解説。2つのタブ切り替え、または上下配置。

### Case 1: TSVから一括でSQLやコマンドを生成（Simple）
> **TSVを貼り付けるだけで、オブジェクト配列として処理**  
> 事前準備なしでループを書き、必要なUPDATEクエリを一気に出力パネルへ生成できます。

```typescript
// resourceに 'users' としてTSVをバインドした場合
for(const row of $resource.users) {
  // 自動パースされ、プロパティ補完が効く
  $println(`UPDATE user_table SET status = 1 WHERE uuid = '${row.uuid}';`);
}
```

### Case 2: 大規模Javaプロジェクトの不参照プロパティ特定（Complex）
> **複数データセットを横断した本格的な解析処理**  
> `.properties` と Javaソースの複数のデータセットを走査。ワークを2工程に分け、中間データを1つのリソース（`$resource.messages`）として保持させながら処理することで、複雑なリファクタリング調査もアプリ内で完結。

```typescript
// Work 1: propertiesファイルから全行を出力し中間データとして保存後...
// Work 2: javaファイルを走査し、不参照キーを絞り込む
const dict = $resource.messages; 
for(const file of $dataset.javaFiles) {
   const content = await file.content();
   // ソースコードとの照合・複雑な集計処理...
}
```

## ✨ 6. クリーンな環境（No Clutter）
**レイアウト**: 左にテキスト、右にクリーンなワークスペースのアイコン。

**コピー**:
> **ストレージを散らかさない、完全なサンドボックス**  
> 
> `package.json` やコンパイル後の `.js` ファイルなどが、あなたのローカルディレクトリに出力されて散らかることはありません。すべてのコードと状態は、一つのワークスペースファイル（`.trk`）内にクリーンに保たれます。
